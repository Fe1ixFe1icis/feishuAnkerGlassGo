核心观点

AI 原生产品创新方法的三个范式转移

传统方法依赖个人经验与事后验证，AI 原生方法通过多智能体分工、证据链约束和对抗前置，把产品创新从「艺术」变成「可审计、可复现、可迭代的工程」。

01

从「直觉驱动」到「证据驱动」

每个关键判断必须挂住可核查信源，AI 推理必须标注置信度，避免把猜测包装成事实。

查看论据
  • 传统方法依赖产品经理经验，如「我觉得用户需要这个功能」，缺乏可追溯依据
  • AI 原生方法要求每个 claim 必须关联 fact（可验证事实）和 source（信源链接）
  • 通过 Pydantic 模型强制约束：EvidenceItem(claim, fact, inference, source, confidence)
  • 示例：「2025 年全球 AI 眼镜出货 870 万台」必须标注来源 Omdia 2025Q1 报告
02

从「后期验证」到「早期对抗」

最凶狠的攻击发生在概念一页纸阶段。用户替身与专家委员会提前暴露风险，而不是等产品原型出来再返工。

查看论据
  • 传统验证发生在原型/样机阶段，返工成本高昂
  • AI 原生方法在概念一页纸阶段就引入红队攻击与专家委员会评审
  • 每个攻击必须基于数据驱动的 persona 和真实场景，未回应的攻击强制改写方案
  • 结果：失败成本前置，避免在错误方向上投入工程资源
03

从「人做脏活」到「人做判断」

AI 负责信息广度与模拟推演，人类只负责高价值决策：选择机会、裁决概念、最终背书。

查看论据
  • AI 负责信息检索、竞品拆解、persona 模拟等广度工作
  • 人类从执行者转变为决策者：选择机会、裁决概念、最终背书
  • 每个阶段设置 human-in-the-loop 检查点，拥有否决权
  • 核心价值：把人的认知资源集中在不可替代的价值判断上
S0 → S6

六阶段对抗式工作流

每个阶段都有明确输入、输出与人类检查点。流程是固定的，但每一次输入的问题与证据都可以不同。

Seven Roles

七角色分工，记忆隔离

每个角色只承担一项职责，只接收上游阶段输出，像流水线一样隔离记忆，避免 AI 说服 AI 的回声室。

S0

提问教练

QuestionCoach

帮助人类把模糊想法变成清晰的问题定义书，禁止 AI 在问题不清时直接给答案。

S1

情报分析师

IntelligenceAnalyst

四路并行检索公司、赛道、用户、竞品信息，每条关键判断挂住可核查信源。

S2

机会侦探

OpportunityDetective

从情报地图中识别矛盾、断裂与空白信号，用五维度量化打分并输出置信度。

S3

概念设计师

ConceptDesigner

基于选定机会强制产出多形态候选概念，禁止在第一阶段直接收敛到唯一方案。

S4

红队

RedTeam

扮演多个数据驱动的用户替身，以第一人称对候选概念发起攻击并给出修正建议。

S5

专家委员会

ExpertCommittee

从技术、供应链、合规、市场四个视角独立评审，输出 pass / revise / reject 裁决。

S6

书记官

Scribe

综合 S0-S5 全部输出，生成结构化产品提案，人类做最终价值背书。

H0-H6

人类仲裁者

Human-in-the-loop

在每个阶段设置检查点，拥有通过、打回、注入约束、最终背书的最高权力。

Core Mechanisms

四大机制，让方法可复现

方法的价值不在于某一次输出有多好，而在于每一次都能稳定地复现这套思考框架。

🔗

Evidence Chain 证据链

每个关键判断强制包含 claim / fact / inference / source / confidence，确保事实与推测分离。

class EvidenceItem(BaseModel): claim: str fact: str | None inference: str | None source: str | None confidence: float
🧠

Memory Isolation 记忆隔离

每个 agent 只接收上游 stage 的 artifact，prompt 中禁止访问其他 agent 的中间结论，避免回声室。

# agent prompt 你只能基于 IntelligenceReport 中的内容进行分析。 不得访问其他 agent 的内部推理、草稿或中间结论。
⚔️

Red Team 对抗验证

概念生成后必须经受用户替身攻击。每个 persona 基于真实数据，提出第一人称质疑，未回应的攻击必须改写方案。

for persona in personas: attack = red_team.attack(concept, persona) concept = revise(concept, attack)
👤

Human-in-the-loop 人类检查点

全自动只是演示模式，真实使用场景下人类在每个阶段都有否决权。AI 做广度，人做判断。

builder.add_node("human_review", human_node) builder.add_edge("human_review", next_stage)
AI Native vs Traditional

这不是「更快」，而是不同

对比传统经验驱动方法，AI 原生方法在信息来源、决策依据、验证方式上都有本质差异。

维度传统方法AI 原生方法
信息获取用户反馈、销售数据、焦点小组四路并行情报检索，单日完成并标注信源
机会判断产品经理嗅觉 + 高管直觉信号聚类 + 五维度量化打分
概念生成内部脑暴，受经验边界限制AI 多形态发散，强制不首次收敛
概念验证评审会，资历权重主导用户替身红队 + 四视角专家委员会
决策依据隐式、难追溯显式 evidence 链，可审计可挑战
典型周期月级单日跑通框架，次日完成修订
核心优势依赖个人认知边界证据链质量 + 提问质量，两者都可迭代
Run It Yourself

Clone 下来，跑一遍

这套工作流不是概念，而是可运行的代码。支持 OpenAI / Anthropic，CLI 与 Streamlit 两种交互模式。

# 1. 克隆仓库
git clone https://github.com/Fe1ixFe1icis/feishuAnkerGlassGo.git
cd feishuAnkerGlassGo
# 2. 安装依赖（推荐 Python 3.10+）
python -m venv .venv
.venv\Scripts\activate
pip install -e ".[dev]"
# 3. 配置 API Key
copy .env.example .env
# 编辑 .env，填入 OPENAI_API_KEY 或 ANTHROPIC_API_KEY
# 4. 全自动跑通 S0→S6
py -m glassgo_workflow.main --auto
# 5. 交互式 Streamlit 界面
py -m streamlit run streamlit_app.py
# 6. 运行测试
pytest tests/ -v
